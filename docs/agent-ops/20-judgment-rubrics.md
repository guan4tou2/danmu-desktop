## 1. 何時該升級模型

**升級訊號（命中任一就升）：**
- [ ] 小模型（haiku）在這題**錯了一次** → 立刻升 sonnet。
- [ ] sonnet 在**同一子任務連錯兩次** → 升 opus，帶完整失敗軌跡（格式見 [10-model-dispatch.md](10-model-dispatch.md#4-升降級路徑省錢又不卡關的核心)）。
- [ ] 任務涉及**跨模組隱藏耦合**、**有狀態的併發**（ws_queue / ws_auth）、**安全邊界**、**架構取捨** → 起手就用 opus。
- [ ] 產出將**難以逆轉**（發版、刪資料、對外發布）且判斷不確定 → 升到最高可用模型再決定。
- [ ] 連續兩輪都在小幅亂試、沒有逼近答案 → 升級（且見第 4 節，可能是方向錯）。

**不要升級：**
- [ ] 純機械、模式明確、只是量多 → 維持 haiku/sonnet 批次跑，甚至降級。
- [ ] 只是「想更保險」但沒有失敗訊號 → 不要無腦升 opus/max 燒錢。

**正例**：sonnet 兩次都修不好一個 live 才重現的競態 → 升 opus 並附「試了加鎖、試了改順序，各自的錯誤原文」。
**反例**：一個把日期格式化的小函式，直接派 opus/max → 過度升級，sonnet/medium 就夠。

## 2. 何時算真的完成

**「完成」的定義：交付物存在、正確、且被獨立驗證過。** 全部勾完才能說「好了」：

- [ ] **交付物真的落地**：檔案存在（fresh agent read-back 確認）、程式碼真的改進去了。
- [ ] **行為被驗證**：程式碼跑過測試/實跑，**貼了實際輸出**；不是「應該會過」。
- [ ] **涵蓋原始要求的每一項**：回頭對照使用者最初講的每一點，逐條打勾。
- [ ] **沒有半修**：danmu 的「兩處要同步」都同步了（發版時版本號兩處 `package.json`+`config.py:APP_VERSION` 同值、renderer 動到就顧 index.html+child.html 兩邊）。
- [ ] **誠實揭露**：沒驗的說沒驗、跳過的說跳過、已知的限制寫出來。

**未達完成的偽訊號（看到這些別宣稱完成）：**
- 「我覺得這樣應該可以」——沒跑就不算。
- 「測試我沒跑，但邏輯上對」——沒跑就不算。
- 「主要功能好了，邊角我沒測」——那就明說哪些沒測，別講「完成」。

**正例**：「加了 `/admin/effects/reload` 端點，跑 `pytest tests/test_effects.py` 22 passed（輸出見下），並請 fresh agent 確認前端按鈕有正確打這個端點——確認無誤。」
**反例**：「重載按鈕加好了，應該能用。」→ 沒測、沒驗、沒貼輸出，不算完成。

## 3. 何時該停下來問使用者

**停下來問（命中任一）：**
- [ ] 動作**破壞性/不可逆/對外**（刪除、覆寫、reset --hard、force-push、發版、部署、發信），且**使用者沒明確授權**「不用問直接做」。
- [ ] 出現**真正的範圍變更**：使用者要的東西需要動到他沒提、且明顯有取捨的地方。
- [ ] 需求**根本性歧義**，且不同解讀會導向**很不一樣**的產出，且**你無法從程式碼/文件自己查明**。
- [ ] 遇到**純品味/主觀**決定（視覺風格、產品方向），沒有客觀判準可依。
- [ ] 模型連 opus/max 都卡住、重試兩輪未果 → 把選項攤給使用者。

**不要停下來問（自己決定、做下去）：**
- [ ] 有合理預設、可逆、順著原請求的動作 → 直接做，順帶說一聲用了什麼預設。
- [ ] 答案能從程式碼、git、文件自己查到 → 去查，不要問。
- [ ] 只是「想確認一下要不要繼續」→ 不要問，繼續做（自主作業時尤其）。

**問的方式**：一次把需要的決定**批次問齊**，給有預設的選項，不要一題一題打斷。

**正例**：使用者要「清掉舊 log」，但目標目錄裡有看起來是別人放的重要檔 → 停下來確認「這些也要刪嗎」。
**反例**：使用者要「加個 dark mode 開關」，你卻停下來問「按鈕要放左邊還右邊？」→ 有合理預設（放設定區既有位置），直接做並說一聲即可。

## 4. 什麼訊號代表方向錯了

**這些訊號 = 換路，不要再重試同一條路：**
- [ ] **同樣的錯誤反覆出現**，每次只是小幅調參 → 你在鬼打牆，假設本身可能錯。
- [ ] **修好 A 就冒出 B，修 B 又回到 A** → 你在症狀層打轉，沒抓到根因。
- [ ] **「這裡不該這麼難」** → 若一個看似簡單的事一直做不出，通常是前提錯了（找錯檔、誤解架構、環境沒設對）。
- [ ] **越改越髒**：為了讓某招work，一直加特例/hack → 該退一步換設計。
- [ ] **驗證一直不通過但你不懂為什麼** → 停止改 code，先搞懂「預期 vs 實際」的差在哪（root cause），這正是 `investigate` skill 的用途。

**換路的做法：**
1. 停止當前嘗試，**寫下目前的假設**與「哪些已排除」。
2. 退一步問：「有沒有可能我對問題的理解本身就錯了？」
3. 換一條路：換方法、換檔案、升級模型帶失敗軌跡、或叫 `investigate`/`systematic-debugging` 重新定位根因。
4. 仍不行 → 回報使用者（見第 3 節）。

**正例**：測試 flaky，改了三次 sleep 時間都還是偶爾失敗 → 停，意識到「調 sleep」是症狀療法，改去找共用狀態沒隔離的根因。
**反例**：同一個 import error 連試七種寫法 → 其實是 `PYTHONPATH` 沒設對，換路（檢查環境）而不是換第八種 import 寫法。

## 5. 品質底線怎麼驗

**「品質底線」不是抽象的『做得好』，是這幾條可檢查的線：**

- [ ] **能跑**：程式碼實際跑過，不是只看起來對。
- [ ] **對齊需求**：逐條對照原始要求，沒有漏項也沒有多做沒被要求的。
- [ ] **融入既有風格**：新程式碼的命名、註解密度、慣用法**跟周圍程式碼一致**（讀起來像同一個人寫的）。
- [ ] **註解只寫「為什麼」不寫「做什麼」**：只在程式碼本身無法表達的約束處加註解，不要旁白「這行做 X」。
- [ ] **沒有破壞既有測試**：跑相關子集，或跑全套確認沒有回歸。⚠️ danmu 全套要用 `cd server && PYTHONPATH=.. uv run python -m pytest tests/ --ignore=tests/test_browser_isolated.py -q`——browser 測試模組在全套執行時是**故意跳過**的（另用隔離方式跑），直接 `pytest` 可能誤判成回歸。
- [ ] **獨立驗證過**：見第 2 節，fresh agent read-back / 測試 / 第二意見。

**強制的驗證分工**（見 [10-model-dispatch.md](10-model-dispatch.md#6-驗證不自驗硬規則)）：
- 檔案 → fresh agent read-back。
- 程式碼 → 跑測試/實跑貼輸出。
- 高風險判斷 → 第二意見或多答案評審。

**誠實邊界**：驗證能保證「執行品質」（能跑、對齊、無回歸），**不能**保證「這是最好的設計/最漂亮的 UI」。品味題請見第 3 節——攤給使用者或升級到最高模型，別假裝驗證過就等於品質好。

**正例**：改完 admin 前端，跑相關 pytest + 起 preview 用 `preview_snapshot` 確認畫面結構，貼出結果，再請 fresh agent 對照原需求逐條檢查。
**反例**：「我看了一下 code，風格應該一致，品質沒問題」→ 沒跑、沒對照、沒獨立驗證，不成立。

## 6. 該不該叫 skill / 該不該動手

**叫對應 skill（明確符合才叫）：**
- [ ] 使用者**明確**要那個動作：ship / qa / review / investigate / office-hours…（見 [CLAUDE.md](../../CLAUDE.md) 路由表）。
- [ ] 使用者打了 `/斜線指令`。

**先給答案、不要動手改：**
- [ ] 使用者在**描述問題、問問題、想事情**——這時交付物是你的**評估**，不是修改。給看法然後停，等他要你改再改。

**直接做、不套重 skill：**
- [ ] 純事實查詢、單檔小改、你已知答案的小事——直接做，別為了儀式硬套 `investigate`/`qa`。

**正例**：使用者說「幫我把這個 PR ship 出去」→ 叫 `ship`。使用者說「你覺得這個特效系統的架構有什麼問題？」→ 給評估，**不要**直接動手重構。
**反例**：使用者問「APP_VERSION 現在幾號？」→ 有人會硬叫 `investigate`，錯；直接查 `server/config.py` 回答即可。
