# F · 維護協議：怎麼安全更新這套制度

> 讀者：未來的 Sonnet / Opus / Haiku session。
> 這套制度是活的，會需要更新。但**更新錯了會污染每一個未來 session**，所以有規矩。
> 核心原則：**改動前先備份、明確的自己改、模糊或結構性的先問、每次踩坑都回寫教訓。**

---

## 1. 改動前的鐵律（每次都做）

1. **先備份**：改任何既有制度檔前，先 `cp X.md X.md.bak`（若已有 `.bak`，用 `X.md.bak2` 之類不要覆蓋舊備份）。
2. **先讀全文**：`Read` 整個要改的檔，理解上下文再改，不要盲改。
3. **改完驗證**：用 fresh agent read-back 確認改動落地且沒有製造矛盾（見 [10-model-dispatch.md](10-model-dispatch.md#6-驗證不自驗硬規則)）。
4. **一次一個主題**：不要在一次維護裡同時大改多個檔，出錯難回溯。

---

## 2. 哪些可以自己改（不用先問使用者）

只要遵守第 1 節鐵律，這些**可以自主更新**：

- ✅ **修正明顯錯誤**：路徑寫錯、工具名打錯、指令過時、連結壞掉。
- ✅ **補一個新踩坑教訓**（見第 4 節格式）到既有檔的對應段落。
- ✅ **補一個新的正例/反例**讓某條判準更清楚。
- ✅ **更新 danmu 事實**：版本號、新結構、新指令——但**先確認新事實是真的**（查程式碼，不是憑印象）。
- ✅ **精簡過長內容**（見第 5 節），前提是**不刪掉任何一條規則的實質**，只是濃縮表達。

---

## 3. 哪些動之前要先問使用者

這些**改動前先問**（因為會改變制度的骨架或預設行為）：

- ⚠️ **改預設模型/取向**：如把主力從 opus 改成別的、把「平衡」改成「省成本」。這是使用者當初拍板的（見本 session 決策），不要自行推翻。
- ⚠️ **刪掉或反轉一條規則**：不是修正錯字，而是「這條規則我覺得不對想拿掉」。先問。
- ⚠️ **改變檔案結構**：新增/合併/刪除制度檔、改 `CLAUDE.md` 的路由骨架。
- ⚠️ **放寬安全/破壞性相關規則**：任何讓破壞性操作更容易的改動，一律先問。
- ⚠️ **改 `.claude/settings.local.json` 權限或加 hooks**：見 [90-hook-suggestions.md](90-hook-suggestions.md)——hook 建議清單在那，但**實際寫入 settings 前要問使用者**（使用者本 session 選擇「只要建議清單」）。

判準：**「這是修正，還是改主意？」** 修正 → 自己改。改主意 → 先問。

---

## 4. 踩坑後：教訓寫回哪、什麼格式

**每次踩到一個新坑（尤其是浪費 token / 跑偏 / 翻車），把教訓沉澱下來，不要讓下一個 session 重踩。**

**寫去哪：**
| 坑的性質 | 寫進 |
|---|---|
| 這 harness 的通用陷阱（token/失焦/出錯） | [00-harness-diagnosis.md](00-harness-diagnosis.md) 對應的三類之一 |
| 模型調度、派工、驗證的教訓 | [10-model-dispatch.md](10-model-dispatch.md) |
| 「什麼時候該怎麼判斷」的教訓 | [20-judgment-rubrics.md](20-judgment-rubrics.md)，補成新判準或正反例 |
| danmu 專案特有的事實/陷阱 | 全域 auto-memory（`MEMORY.md` 及 `memory/`，非 repo 檔），不是這裡 |
| 只跟這次任務有關、之後用不到 | **不要寫**進制度，寫進當次 session 的產出即可 |

**格式**（附加在對應段落，維持該檔既有風格）：
```
- **症狀**：<看到什麼>
- **為什麼**：<根因>
- **修法**：<具體怎麼做，可勾選>
- 正例：<一句> / 反例：<一句>
```

**判準：值不值得寫進制度？** 問三個問題，全「是」才寫：
1. 這個坑**會重複發生**（不是一次性偶發）嗎？
2. 它**跨 session 適用**（不是只有這次任務）嗎？
3. 有**具體可執行的修法**（不是空話「要小心」）嗎？

**反例**：不要寫「要保持高品質」「要仔細」這種抽象話——對弱模型等於沒寫（見 [50-letter](50-letter-to-future-sessions.md) 的退化警告）。

---

## 5. 精簡：累積多長要瘦身

**制度檔會膨脹，膨脹到弱模型讀不完就失去意義。** 觸發精簡的訊號：

- 單一制度檔超過 **~250 行**，或出現**明顯重複**（同一規則在兩處講）。
- `MEMORY.md` 超過 **~150 行**，或塞了一堆過時的 session 筆記。
- 同一段有 **3 個以上**在講同一件事的正反例——留最清楚的 1~2 個。

**精簡怎麼做（安全版）：**
1. 先備份（第 1 節）。
2. **只濃縮表達，不刪規則實質**：把囉嗦句改精煉，把重複合併，把過時的刪掉。
3. 每刪一段，問：「這條規則的實質有沒有在別處保留？」沒有就不能刪。
4. 過時的 session 記憶 → 可搬到 `memory/` 的 archive 或直接刪（那些是歷史，不是規則）。
5. 精簡後 fresh agent read-back：確認「規則數量沒少、只是變短」。

**精簡的黃金比例**：`00`~`20`（診斷、調度、判斷）是核心，要**留白給強模型、明確給弱模型**——寧可精煉也不要為了短而砍掉判準與範例。`30`（模板）可以長，因為是拿來複製的。

---

## 6. 這套制度的「不變量」（維護時不能破壞的底線）

無論怎麼更新，這幾條是地基，動它們一定先問使用者：

1. **指揮官不下場**（大量讀取/掃描/批改派 subagent）。
2. **驗證不自驗**（fresh agent / 測試 / 第二意見）。
3. **破壞性操作先確認**。
4. **改既有檔先備份**。
5. **抽象要求不算數**——每條規則都要具體、可執行、有判準或範例。
6. **CLAUDE.md 只放路由與硬規則**，細節在 `docs/agent-ops/`。

若哪天發現這些不變量本身有問題，那是**跟使用者討論的層級**，不是單一 session 自行改。
