# C · 模型調度守則

> 讀者：未來的主對話模型（預設 **Opus 4.8**），以及它派出去的 subagent。
> 目標：讓「一顆較弱的主模型 + 一群 subagent」的組合，產出接近強模型的品質，而不燒爆 token。
> 取向：**平衡**（Sonnet 是主力打工模型，必要才升 Opus，可重複的批次降 Haiku）。

---

## 0. 一句話心法

**指揮官不下場。** 主對話的角色是「決策 + 收斂結論」，不是「親自搬磚」。大量讀取、掃 repo、查網頁、批次改檔——一律派 subagent，主對話只收**結論**與**`檔案:行號`**。主對話的 context 是稀缺資源，用來裝判斷，不是裝原始檔案。

為什麼：見 [00-harness-diagnosis.md](00-harness-diagnosis.md) A1。指揮官每親自吞一份原始檔，就永久佔用 context 到 session 結束，且會逐漸忘記任務。

---

## 1. 指揮官不下場：什麼一定要派出去

| 這類工作 | 一定派 subagent | 用哪種 agent |
|---|---|---|
| 掃很多檔 / 不確定在哪 / 跨檔找命名 | ✅ | `Explore`（唯讀、只回結論） |
| 讀長檔只為抓幾個事實 | ✅ | `Explore` 或 `general-purpose` |
| 研究一個主題 / 查網頁 / 讀 API 文件 | ✅ | `general-purpose`（給它 WebSearch/WebFetch/context7） |
| 批次改很多檔（同一模式套 N 處） | ✅ | `general-purpose`，模式明確時降 Haiku 批次跑 |
| 需要平行做的獨立子任務（2 個以上互不相依） | ✅ | 一則訊息內開多個 agent 並行 |
| 驗證產出（read-back / 跑測試 / 第二意見） | ✅ | **fresh-context** agent（見第 6 節） |

**可以親自做**（不必派）：單檔的針對性 `Read`（已知路徑、要哪段）、單檔小 `Edit`、跑一條測試指令、回答已在 context 裡的事實、最終對使用者的說明。

判準：**「這會不會把大量原始內容灌進我的 context？」** 會 → 派出去。**「這是不是需要一個沒有我當前假設的乾淨視角？」** 是 → 派 fresh agent。

## 2. 派工三件套（每次派 subagent 都要寫齊）

subagent 看不到你的對話，只看到你給它的 prompt。**缺一件，它就會亂猜。** 每則派工 prompt 必含：

1. **目標與動機**：要什麼、為什麼要。動機讓它在遇到分歧時能自己判斷。
   - 反例：「看一下 effects.py」→ 太模糊。
   - 正例：「我要在 admin 加一個『重載特效』按鈕，需要知道後端重載的入口函式與它的 API 路由，這樣前端才知道打哪個端點。」
2. **驗收條件**：怎樣算做完、輸出要滿足什麼。可量化就量化。
   - 正例：「回報：(a) 重載函式的 `檔案:行號`，(b) 對應的 HTTP 路由與 method，(c) 有沒有現成測試涵蓋它。三項都要有。」
3. **回報格式**：要它回什麼形狀（見第 5 節回報合約）。
   - 正例：「用三行 bullet 回，每行 `檔案:行號 — 一句說明`。不要貼整段程式碼。」

完整可填空模板見 [30-delegation-templates.md](30-delegation-templates.md)。

## 3. 顯式指定 model 與 effort

**每次派 agent 都明講 `model` 與 `effort`，不要靠預設猜。** 這個環境可用的模型（Agent 工具 `model` 參數）：`haiku` / `sonnet` / `opus` / `fable`。effort：`low` / `medium` / `high` / `xhigh` / `max`。

成本由低到高：**haiku < sonnet < opus < fable**。（`fable` 是最高階，可能不一定在；在的話留給最難的品味 judgement，見升降級路徑。⚠️ Agent 工具 `model` 只接受這四個值——`mythos` 之類不是合法值，填了會 InputValidationError。）

### 任務 → 模型 對照（平衡取向）

| 任務性質 | 預設 model | 預設 effort | danmu 例子 |
|---|---|---|---|
| 機械性、模式明確、可批次 | `haiku` | `low`~`medium` | 把 20 處 `console.log` 改成統一 logger；套一個已解出的 rename 到多檔 |
| 一般搜尋 / 讀檔抓事實 | `sonnet` 或 `Explore` | `medium` | 「找出所有註冊 admin 路由的地方」 |
| 一般實作 / 一般 bug 修 | `sonnet` | `medium`~`high` | 加一個 admin 端點；修一個明確的前端事件 bug |
| 較難實作 / 跨模組 / 有隱藏耦合 | `opus` | `high` | 改 renderer（記得 index+child 兩邊）；動 ws_queue/ws_auth 這種有狀態的東西 |
| 難 bug / 症狀與原因不明 | `opus` | `high`~`xhigh` | 「測試偶爾 flaky」「live 環境才出現的競態」 |
| 高風險判斷 / 架構取捨 / 品味 | `opus`（不夠再 `fable`） | `xhigh`~`max` | 「這個 IA 重構值不值得」「兩個設計方案選哪個」 |
| 驗證用 fresh agent | 至少等於**被驗對象**的等級 | `medium`~`high` | read-back 用 `sonnet`；驗高風險判斷用 `opus` |

規則：
- **不確定難度 → 從 `sonnet`/`medium` 起手**，錯了再升（見第 4 節），不要一開始就 `opus/max` 燒錢，也不要用 `haiku` 硬扛難題。
- **effort 要配任務**：純機械用 `low`（省）；要推理的別低於 `medium`；最難的品味/驗證才 `xhigh`/`max`。
- **平行批次同模式**：一旦某個模式被 `opus` 解出來，把「怎麼套」寫清楚，用 `haiku`/`low` 平行套到其餘 N 處（降級批次，見第 4 節）。

## 4. 升降級路徑（省錢又不卡關的核心）

| 情況 | 動作 |
|---|---|
| 小模型（haiku）**錯一次** | **直接升一級**（→ sonnet）。不要讓 haiku 重試同一題。 |
| 中階模型（sonnet）**同一子任務連錯兩次** | **升到 opus**，且**附上完整失敗軌跡**（兩次都試了什麼、各自怎麼錯、錯誤訊息原文）。 |
| opus 也卡住（同一子任務再連錯兩次） | 停止重試。要嘛 `fable`（若可用）＋`max` 打一次；要嘛**回報使用者**這題超出自動化能力（見 [20-judgment-rubrics.md](20-judgment-rubrics.md)「何時該停下來問」）。 |
| 某模式被高階模型**解出來了** | **降回便宜模型批次套用**（把解法寫成明確步驟給 haiku/sonnet 平行跑）。 |
| 同一件事 | **最多重試兩輪**。兩輪未果就升級或求助，不要無限鬼打牆。 |

**升級時一定要帶失敗軌跡**——不然升上去的模型會從零重踩同樣的坑。失敗軌跡格式：
```
子任務：<一句話>
嘗試 1（model/effort）：做了什麼 → 結果/錯誤原文
嘗試 2（model/effort）：做了什麼 → 結果/錯誤原文
目前卡在：<具體卡點>
已排除：<確定不是這些原因>
```

**正例**：haiku 改一個 regex 改錯 → 立刻換 sonnet 重做，不糾纏。
**反例**：讓 haiku 對同一個失敗的測試連改五次，每次都小幅亂試 → 燒 token 又不會過，應該錯一次就升。

## 5. 回報合約（subagent 要怎麼回）

subagent 的最終訊息**只回到主對話**，使用者看不到。所以：

1. **只回結論與依據**，不要複述過程、不要貼大段原始檔。
2. **指位置用 `檔案:行號`**，讓主對話能點開，不用把內容搬回來。
3. **長產物（報告、大段程式、清單）落檔**：寫進 `docs/agent-ops/` 之外的暫存（如 scratchpad 或指定檔），**只回檔案路徑 + 3~5 行摘要**。
4. **結構化資料用固定形狀**：能用 schema 就用 schema（Agent 的結構化輸出），主對話不用再 parse。
5. **明確標註信心與缺口**：哪些是確定的、哪些是推測、哪些沒查到。**不要**用「應該」「大概」假裝確定。

主對話收到後：**把關鍵結論寫進自己的回覆/todo**，因為 subagent 的原文不會自動變成使用者看得到的東西。

## 6. 驗證不自驗（硬規則）

**寫產出的那顆模型，不驗自己的產出。** 驗證一律派**fresh-context** agent（沒有你當前假設的乾淨視角）。按產出類型選驗法：

| 產出類型 | 驗法 | 具體做法 |
|---|---|---|
| 檔案 / 文件 | **read-back** | fresh agent `Read` 該檔，回報「是否存在、是否完整、有沒有壞連結/矛盾」。 |
| 程式碼改動 | **測試或實跑** | 跑 pytest / 起 preview / 實際觸發該路徑，**貼實際輸出**。不是「應該會過」。 |
| 高風險判斷 / 品味 | **第二意見或多答案評審** | 另開 1~2 個 agent 獨立給結論；分歧就攤開比較，或用「多答案 → 評審選優」。 |
| 事實宣稱 / 研究結論 | **對抗式查證** | fresh agent 專門去「反駁」這個結論，反駁不掉才算成立。 |

驗證 agent 的 model：**至少等於被驗對象的等級**。用 haiku 去驗 opus 的架構判斷沒有意義。

誠實條款：拆解 + 驗證 + 多樣本評審能補**執行品質**（找漏、抓錯、確認落地）。但**模糊題與品味判斷補不了**——遇到「這樣設計好不好看」「這個取捨對不對」這種，驗證只能縮小範圍、不能給定論。此時：升級到最高可用模型打一次、找外部第二意見（使用者或另一工具如 `codex` skill）、或**明說這超出自動判斷、把選項攤給使用者**。詳見 [20-judgment-rubrics.md](20-judgment-rubrics.md)。

## 7. 反面清單（別這樣調度）

- ❌ 主對話自己 `Read` 三支大檔找一個函式名 → 該派 Explore。
- ❌ 派工只寫「看一下 X」→ 缺驗收條件與回報格式，subagent 會亂回。
- ❌ 全部用 `opus/max` 打 → 燒錢；機械活該降 haiku。
- ❌ 全部用 `haiku` 硬扛 → 難題卡死；該升就升。
- ❌ 讓同一顆模型驗自己剛寫的東西 → 系統性放過自己的錯。
- ❌ 升級卻不帶失敗軌跡 → 新模型從零重踩。
- ❌ 同一題重試五次以上 → 早該升級或求助。
