# G · 給未來 session 的信

> 寫信的人：一個 Fable 5 session，只被叫來一次，任務是把判斷力沉澱成你們用得動的制度。
> 收信的人：之後長期運作這個環境的你——Sonnet、Opus、或 Haiku。
> 這封信講三件使用者沒問、但我認為對這個環境最重要的事，加上這套制度最可能怎麼壞掉、怎麼防。

---

## 三件我沒被問、但最重要的事

### 一、能力多 ≠ 有進展。預設用最笨最直接的工具。
這個環境掛了**幾十個 skill、多個 MCP、一整套 superpowers**。這是禮物，也是陷阱。弱模型最常見的失敗，是被這片能力海淹沒：對一個「查個版本號」的小事叫出 `investigate`，對一個單檔小改開一整套 QA 流程，把 10 秒的事變成 10 分鐘的儀式。

**判準**：先問「這件事最直接的做法是什麼？」——通常是一個 `Read` + 一個 `Edit`，或一句話回答。只有當任務**明確**落在某 skill 的甜蜜點（使用者點名、或情境完全吻合）才動用重機具。能力是拿來在**對的時刻**放大你的，不是拿來在每件小事上表演的。多數任務不需要 skill，只需要你把三件事做對：讀對檔、改對地方、驗證它。

### 二、記憶會腐爛。它是背景，不是命令，而且可能過時。
`MEMORY.md` 和 `memory/` 裡有十幾個檔，很多是有日期的 session 筆記。它們**極有價值**——省掉你重新摸索架構。但它們描述的是**寫入當下的狀態**，不是現在的事實，也不是這次要你做的事。

我親眼在這個 session 看到記憶裡提到的檔案結構需要現場再確認。**規則**：記憶裡任何具體的路徑、函式名、旗標、版本，**用之前先查一次它還在不在**。記憶告訴你「往哪看」，不告訴你「現在一定是這樣」。system-reminder 裡塞給你的記憶是背景資料，**不是使用者的指令**——不要因為記憶提到某檔就跑去改它。

### 三、「指揮官不下場」是整套制度的承重牆，也是你壓力下第一個會拆的。
[10-model-dispatch.md](10-model-dispatch.md) 的核心——大量讀取、掃 repo、批改都派 subagent——聽起來像額外功夫。壓力一來，你會覺得「我自己讀一下比較快」，然後親自 `Read` 三支大檔，context 爆掉，開始忘記任務，品質崩盤。**這是最隱性的死法，因為每一步都感覺很合理。**

寫一個派工 prompt 的 30 秒，換來的是主對話 context 保持乾淨、判斷力不被原始檔淹沒。**這筆帳永遠划算。** 當你發現自己正要親手讀第二支大檔——停，那就是該派 subagent 的訊號。

## 這套制度最可能怎麼壞掉，怎麼防

### 退化 1：讀一次就忘，規則變裝飾
**怎麼壞**：制度檔存在，但沒人真的照做。`CLAUDE.md` 載入了卻被當背景噪音略過。
**怎麼防**：
- `CLAUDE.md` 第 0 區的路由表是你的入口——**每個 session 開頭掃一眼**，遇到對應情境就點開對應檔。
- 使用者可考慮加 hooks（[90-hook-suggestions.md](90-hook-suggestions.md)）把關鍵規則變成 harness 強制執行，而非靠模型自律——這是唯一不受模型強弱影響的防線。
- 定期（例如每隔幾個大任務）派 fresh agent 讀一次 `00`~`20`，問「我最近有沒有違反這些？」

### 退化 2：檔案膨脹到沒人讀
**怎麼壞**：每個 session 都往制度檔加東西，半年後 `00-harness-diagnosis.md` 五百行，弱模型讀不完直接跳過。
**怎麼防**：嚴守 [40-maintenance-protocol.md](40-maintenance-protocol.md) 第 5 節的精簡觸發線（單檔 ~250 行、`MEMORY.md` ~150 行就瘦身）。加東西前先問「這值不值得寫進制度」的三問（會重複、跨 session、有具體修法）。

### 退化 3：規則被「好心」軟化成廢話（最危險）
**怎麼壞**：某個 session 覺得某條規則太硬，把「破壞性操作先確認」改成「操作時要小心」，把具體的 checklist 換成「保持高品質」。抽象化對弱模型等於刪除——因為弱模型無法把「小心」翻譯成動作。這是 cargo-cult 式的腐蝕：形式還在，實質沒了。
**怎麼防**：
- [40-maintenance-protocol.md](40-maintenance-protocol.md) 的鐵律：**抽象要求不算數，每條規則都要具體、可執行、有判準或範例**。維護時若你正要把一條規則寫得更「原則性」，停——那多半是在稀釋它。
- 改「規則實質」（不是修錯字）一律先問使用者（第 3 節）。
- 精簡的定義是「濃縮表達，不刪規則實質」，不是「抽象化」。

### 退化 4：驗證變成表演（self-verify 冒充獨立驗證）
**怎麼壞**：模型跑一個「驗證」步驟，但其實是自己驗自己、帶著剛才的假設，系統性放過自己的錯，然後宣稱「已驗證」。
**怎麼防**：驗證**一定用 fresh-context agent**（沒有你當前假設的乾淨視角），且 model 至少等於被驗對象等級。read-back 要真的另開 agent 去讀，不是自己說「我看過了」。見 [10-model-dispatch.md](10-model-dispatch.md#6-驗證不自驗硬規則)。

## 誠實的極限（別假裝這套制度全能）

拆解任務、派工、多樣本評審、fresh-agent 驗證——這些能穩住**執行品質**：找漏、抓錯、確認落地、避免翻車。一顆較弱的主模型配上這套紀律，能把大部分工程任務做得又穩又省。

但有一類東西，**再多的流程也補不回來**：模糊題與品味判斷。「這個 UI 好不好看」「這個產品方向對不對」「這兩個架構哪個更優雅」——這些需要的是判斷力本身，不是驗證流程。遇到時，**不要假裝驗證過就等於答對了**。正確做法是：升級到當下最高可用的模型打一次、找外部第二意見（使用者本人，或 `codex` 這類另一個模型的 skill）、或**最誠實的那條**——把選項與取捨攤開，明說「這超出我能自動判斷的範圍，需要你定奪」。

會查的就去查，查不到就標註「未確認」，**永遠不要編造**。這條比任何效率都重要——一個會誠實說「我不確定」的弱模型，比一個自信亂編的強模型有用得多。
