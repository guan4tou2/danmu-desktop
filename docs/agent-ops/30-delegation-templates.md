# E · 派工 Prompt 模板

> 讀者：未來的主對話模型。要派 subagent 時，**複製對應模板、把 `<...>` 填滿再送**。
> 每個模板都內含「派工三件套」（目標與動機 / 驗收條件 / 回報格式，見 [10-model-dispatch.md](10-model-dispatch.md#2-派工三件套每次派-subagent-都要寫齊)）。
> 送出時記得同時指定 `model` 與 `effort`（見各模板末的建議）。subagent 看不到你的對話，模板裡要自帶所有背景。

通用原則（所有模板都適用）：
- subagent 的回覆只回到你，使用者看不到 → 收到後把關鍵結論寫進你自己的回覆。
- 要它**只回結論 + `檔案:行號`**，長產物落檔回路徑。
- 要它**標註信心與缺口**，不要用「應該」假裝確定。

---

## 1. 搜尋 / 定位（用 `Explore`）

```
目標與動機：
我要 <做什麼>，為此需要在 repo 裡找出 <要找的東西>。這樣我才能 <為什麼需要>。

範圍：
在 <目錄/檔案範圍，如 server/routes/ 或整個 repo> 裡找。
搜尋廣度：<medium 一般 / very thorough 要涵蓋多處與多種命名>。

驗收條件（全部要有）：
- <要點 1，如：所有註冊 XX 路由的位置>
- <要點 2，如：對應的 handler 函式名>
- <要點 3，如：有沒有現成測試涵蓋>

回報格式：
每個結果一行：`檔案:行號 — 一句說明`。
最後補一句總結：<你要它下的結論，如「共 N 處，入口在 X」>。
不要貼整段程式碼，不要複述搜尋過程。查不到的項目明確說「未找到」。
```
建議：`model: sonnet`（或直接用 `Explore` agent），`effort: medium`。範圍大且要多種命名時 `very thorough`。

---

## 2. 實作（用 `general-purpose`）

```
目標與動機：
在 <哪個模組/檔案> 實作 <功能>，目的是 <使用者要的價值>。

背景（subagent 看不到對話，這裡給齊）：
- 相關檔案：<檔案:行號 或路徑清單>
- danmu 硬規則（若相關）：<如：改 renderer 要顧 index.html 和 child.html 兩邊 / 版本號改兩處>
- 既有慣例：<命名、風格、要沿用的 pattern>

實作要求：
- <具體要做的事，逐條>
- 融入既有程式碼風格（命名、註解密度、慣用法一致）。
- 註解只寫「為什麼」，不寫「這行做什麼」。

驗收條件：
- <可觀察的成功條件，如：POST /xxx 回 200 且寫入 YYY>
- 不得破壞既有測試：跑 <相關測試指令，如 cd server && PYTHONPATH=.. uv run python -m pytest tests/test_xxx.py -q>。
- 貼出測試實際輸出。

回報格式：
- 改了哪些檔：`檔案:行號 — 改了什麼`。
- 測試結果：貼實際輸出（passed/failed 數字）。
- 有沒有沒做完/不確定的：明確列出。
```
建議：一般 `model: sonnet, effort: high`；跨模組/有狀態/安全相關 `model: opus, effort: high`。
⚠️ 實作完**不要讓同一個 agent 說「驗好了」就算數**——另派 fresh agent 驗證（見模板 5 或 [10-model-dispatch.md](10-model-dispatch.md#6-驗證不自驗硬規則)）。

---

## 3. 重構（用 `general-purpose`，多檔平行時考慮 `isolation: worktree`）

```
目標與動機：
把 <現況，如：admin.js 裡重複的 fetch 樣板> 重構成 <目標，如：抽成共用 helper>，目的是 <為什麼，如：降低重複、之後好改>。
這是重構——**外部行為必須完全不變**。

範圍與邊界：
- 只動：<允許改的檔案/範圍>
- 絕不動：<不准碰的東西，如：公開 API 形狀、測試斷言>

重構步驟（若已知模式，寫清楚讓便宜模型能照做）：
1. <步驟>
2. <步驟>

驗收條件：
- 行為不變的證據：**重構前後跑同一套測試，結果一致**。跑 <測試指令>。
- 沒有新增未使用的 import / 死碼。
- 貼出測試前後輸出。

回報格式：
- 改動摘要：`檔案:行號 — 改了什麼`。
- 測試前/後結果（證明行為不變）。
- 任何無法在不改變行為下完成的部分：停下來說明，不要硬改。
```
建議：模式明確可批次 → `model: haiku, effort: medium` 平行套；模式要判斷 → `model: opus` 先解一處，再降級批次。多個 agent 同時改檔會衝突時，加 `isolation: worktree`。

## 4. 研究（用 `general-purpose`，給它 WebSearch/WebFetch/context7）

```
目標與動機：
我需要弄清楚 <問題>，用來決定 <這個研究要支撐的決策>。

範圍：
- 要回答的具體問題：<逐條列，越具體越好>
- 可用來源：<repo 內程式碼 / 官方文件（用 context7）/ 網路（WebSearch/WebFetch）>
- 時效性：<若需要最新資訊，明講>

驗收條件：
- 每個結論都要有依據（來源 URL 或 `檔案:行號`）。
- 區分「確定的事實」與「推測」。
- 對抗式自我檢查：對每個關鍵結論，問「有沒有反證？」查得到反證就標註。

回報格式：
- 每個問題一段結論 + 依據來源。
- 一份「信心分級」：高信心 / 中 / 低（附原因）。
- 若結論會超過 ~30 行，落檔到 <scratchpad 路徑>，只回路徑 + 5 行摘要。
```
建議：`model: sonnet, effort: high`；牽涉重大決策 `model: opus`。要深度多源查證時直接用 `deep-research` skill。

## 5. 審查 / 驗證（用 fresh-context agent，**這是驗證不自驗的關鍵**）

```
目標與動機：
有人剛完成了 <什麼產出>。你是獨立審查者，帶乾淨視角——**不要假設它是對的**。

被審對象：
- <檔案路徑 / diff 範圍 / 要驗的宣稱>
- 原始需求（拿來對照）：<使用者最初要的每一點>

審查要求（逐條檢查並回報結論）：
- [ ] 交付物是否真的存在、完整（檔案 read-back / diff 確實存在）。
- [ ] 是否對齊原始需求的**每一項**（逐條打勾或標未達）。
- [ ] 程式碼是否真的能跑：跑 <測試/實跑指令>，貼實際輸出。
- [ ] 有沒有半修（danmu：版本兩處、renderer 兩 HTML、其他同步點）。
- [ ] 有沒有規則互相打架、路徑錯誤、對弱模型模糊的語句（若審的是文件）。

回報格式：
- 判定：PASS / FAIL（有一項未達就 FAIL）。
- 逐條結果：`項目 — 通過/未達 — 依據（檔案:行號 或 輸出）`。
- FAIL 時：具體指出哪裡要修，附 `檔案:行號`。
不要幫忙修，只回報。修由主對話決定。
```
建議：驗證 agent 的 `model` **至少等於被驗對象等級**——驗 opus 的判斷用 `opus`；read-back 文件用 `sonnet`。高風險判斷再開第二個獨立 agent，分歧就攤開比較。
