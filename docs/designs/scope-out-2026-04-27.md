# Scope-out · 2026-04-27

> **產品決策**：以下功能 **不在 Danmu Fire 的 scope** 內。請 Design 在後續 bundle / artboard 中**避免引入這些概念**。
>
> 工程已在 [`design-v2-prototype-gaps-2026-04-27.md`](./design-v2-prototype-gaps-2026-04-27.md) §F + §G 對齊 review 後做出此決定。

## A. Sessions / 場次 entity ❌ NOT IN SCOPE

### A.1 為什麼移除

V1Z4 batch7-8 的多個 prototype（AdminSessionsPage / SessionDetail / CrossSessionSearch / AdminAudiencePage）假設有「場次」這個 entity — 訊息可以歸屬到某場活動，跨場可以搜尋、聚合觀眾。

但 Danmu Fire 的核心定位是**單機、單一活動 / streaming overlay 工具**：

- 部署模型是「主辦人開啟 server → 觀眾 fire 訊息 → 結束 → 也許下次」
- 不是 conferencing platform、不是 SaaS multi-tenant
- 操作員在乎「現在這場」、不是「7 場前的 OKR Town Hall」
- 把 sessions 加進來會：
  - 強迫 schema 大重構（每筆訊息要 `session_id`）
  - 強迫 UI 加「Start Session / End Session」這個 friction step
  - 跟既有「BROADCASTING / STANDBY」拍號（兩者重疊但語義不同，使用者要學兩套）
  - 有 idle session、abandoned session、auto-close 邏輯這類 edge case 要寫
- 場次概念對 99% 使用者是 over-engineering，對需要的 1% 也只是把 spreadsheet logging 推到我們身上

**結論**：訊息 / poll / fire_sources / audit 全部維持 flat timeline；不引入 session_id 欄位；不做 lifecycle。

### A.2 移除後 prototype 的 4 個 page 怎麼辦

| Prototype | 處理方式 |
|-----------|----------|
| AdminSessionsPage（場次列表） | ❌ 不實作。請 Design 不要再畫這個 page。 |
| AdminSessionDetailPage（單一場次完整時間軸） | ❌ 不實作。對應功能：`#/history` 已有「過去訊息匯出」+「重播」tab，足夠。 |
| AdminSearchPage（跨場次搜尋） | ❌ 不實作 cross-session 維度。**改成單一全文搜尋**整合到 `#/history` 或 `#/messages` 列表的搜尋框（已存在）。 |
| AdminAudiencePage（觀眾列表） | 🟡 **變更實作**：去掉「出席場次數」column，改成 fingerprint aggregation across **all-time**（首次出現 / 最後出現 / 總訊息數 / 違規次數）。已可從 fingerprint_tracker 拿到。 |

### A.3 影響的 prototype 區塊

請 Design 在後續修改 prototype 時**移除以下欄位 / 概念**（否則我們不會實作）：

- `session_id` 任何訊息 / poll / 觀眾的 session 欄位
- 「sess_xxxx」這類 ID 顯示
- 「Start Session」/「End Session」按鈕
- 「session 持續時間」/「場次內訊息數」這類 per-session 統計
- 「跨場次」這類描述（直接寫「全部歷史」即可）
- AdminPollDeepDivePage 的 `sess_2d9e` 標籤、Notifications inbox 「事件鏈」中提到的「session 13:30 開始」這種

## B. 多 admin / 角色制 ❌ NOT IN SCOPE

prototype 偶爾出現「admin / kevin」兩個操作者（例：Audit Log 操作者 filter 列出 admin + kevin 兩人）。

**Danmu Fire 是單密碼登入的單機產品，沒有帳號概念。**

請 Design 移除：
- 多 actor 的 avatar 圓圈（Audit Log column）
- 「操作者」filter 群組（permanently 永遠只有 1 個 admin）
- 任何 role / permission / team 的暗示

audit_log 仍然會記 actor 欄位（值固定 "admin"，是給未來預留的，不展示分類 UI）。

## C. SHA-256 audit 簽章 / 90 天保留期 / SIEM forwarder ❌ NOT IN SCOPE（v1）

prototype Audit Log 提到「保留 90 天 · 每筆 SHA-256 簽章」。這暗示企業級 compliance 需求。

我們 v1 的決定：
- **保留期**：用磁碟 size rotation（2 MiB → `.1`backup），不用時間。Append-only 已防竄改 80%，再加簽章是 over-engineering。
- **沒有 SIEM 推送 / Slack alert webhook for audit events** — 跟 Notifications inbox 區隔開，inbox 是「現場警示」、audit 是「歷史紀錄」，不重複。

請 Design 在 Audit Log artboard：
- 移除 "保留 90 天 · 每筆 SHA-256 簽章" 這行 hint
- 移除 "推送到 SIEM" / "Slack alert" 提示
- 改成 "保留至磁碟 rotation（2 MiB）· 純本機 append-only"

## D. Geo / IP geolocation ❌ NOT IN SCOPE（v1）

prototype Poll Deep-Dive、Audience、Notifications 都有「TW / JP / US / 其他」這類國家分佈。

我們 v1 的決定：
- 不引入 GeoIP service（MaxMind / ipinfo / etc.）
- 不下載 GeoIP database（會增加 image size 200 MB+ 或需外部 API）
- IP 仍然會記錄（fire_sources 已記），但不做 lookup

請 Design 移除：
- Geo bar chart（4-row TW/JP/US/其他）
- 跨選項地區交叉（geo × option matrix）
- 任何「（TW · Taipei）」這類城市標籤
- 「匿名 VPN 偵測」這類需要 GeoIP intelligence 的功能

如果 Design 之後想加，先確認業務需求（誰會看？決策什麼？），再考慮接外部 service。

## E. Pro Edition / License / 收費模式 ❌ NOT IN SCOPE

prototype About page 有「Pro Edition · 訂閱 · 客戶 ID · 到期日」這套 license 卡。

**Danmu Fire 是 MIT 開源、無收費版本。**

請 Design 移除：
- About 的 License section（保留簡單版："License: MIT"）
- 任何「Pro / Free / Enterprise」對比
- 「組織 / 授權人數 / 同時場次 N」這類限額顯示
- 「客戶 ID cust_xxxx」

## F. Sentiment Index / 情感分析 ❌ NOT IN SCOPE

prototype Poll Deep-Dive 有「+47 sentiment / 正面 - 負面」。

**我們不做 sentiment ML / NLP**。
- v5.x 不引入 NLP model（會增加部署成本 + 算錢）
- 但「+1 求簡報 是正面 / 退費 是負面」這類靠 emoji + keyword 的 cheap heuristic 是可以做的（純 FE，看作粗糙版）— 留到 G8 polish 工作裡，但不是 ML

請 Design 標清楚這是 heuristic-based，不是 ML / 不需要訓練資料 / 不需要外部 API。
